# サイドミッション：Blender ベース「直感で伝わる」デモ

## 位置づけ

これはメインロードマップ（`.local/roadmap.md`）の**支線**であり、製品機能ではない。
目的はただ一つ：

> Phase 2 までの成果を、専門知識のない人が見て
> 「透明／不透明の混ざったオブジェクトの外観がちゃんと描けてきているな」
> と直感的に感じられる “作品ショット” を、繰り返し安定して作れるようにする。

断面図・フィールド可視化など専門家向けの診断はここでは扱わない（必要になれば別支線）。
光学係数は暫定・未校正であり、このデモは**定性的な見た目のデモであって物理予測ではない**。
すべての出力に「provisional / uncalibrated / qualitative demo」と明示する。

## なぜ Blender か

普通の人が「透明だ」と感じるのは、オブジェクト単体ではなく**環境との関係**を見たとき：

- 向こうにある花瓶や模様が、オブジェクト越しに**歪んで透けて見える**（屈折）
- ランプの光がオブジェクトの中で**滲む・拾われる**（散乱・吸収）
- 床に**接地影**が落ちて、そこに「在る」感じが出る

これらは “きれいな環境アセット（部屋・HDRI・ランプ・小物）” があって初めて成立する。
Blender/Cycles はこの手の既製アセットとライティングが圧倒的に扱いやすいので主役に据える。

## 技術的な前提とハイブリッド構成

現状の `export openvdb` は次を書き出している（`src/vbdmat/exporters/openvdb.py`）：

- `sigma_a_{r,g,b}` / `sigma_s_{r,g,b}` … RGB 成分グリッド
- `cycles_absorption` / `cycles_scattering` … Cycles 用のスカラー簡約グリッド
- `ior` … 屈折率グリッド（**VDB には入っているが、現行の Cycles プルーフではノードに未接続**）

したがって「向こうが歪んで透ける」屈折効果を出すには、ボリューム単体では足りない。
**ハイブリッド・マテリアル**を組む：

- **外表面** = Glass/Refraction BSDF。IOR は VBDMAT のエクスポート値（透明レジン 1.48 等）を使用
- **内部** = Volume Absorption + Volume Scatter。密度は `cycles_absorption` / `cycles_scattering`
  （色を出したい場合は RGB 成分グリッドから合成）

これは Cycles でガラス質ボリュームを描く標準的なやり方であり、
屈折（透ける歪み）と、透明／不透明の混ざり具合の両方を一枚で表現できる。

### 検証で判明した重要事実（2026-07-02 の実測）

Docker（`vbdmat-openvdb-cycles`）で徹底検証した結果、現行の黒画像は
**単一のコードバグではなく、次の3要因の複合**だと確定した：

1. **透明テストピース × Cycles の IOR 界面省略**（＝本質）。
   `stepped_wedge` は `cycles_scattering` が全ゼロ・吸収 1.167 m⁻¹ のみ、すなわち
   **完全に透明レジンだけ**でできている。`window_coupon` も大半が透明。
   透明媒質は散乱も吸収もほぼ無く、Cycles は内部 IOR 界面を落とすため、
   参加媒質“だけ”では物理的にほぼ不可視になる。Mitsuba が段を描けるのは界面メッシュを
   作っているからで、Cycles で見えないのは想定どおりの帰結。
2. **カメラ near クリップがデフォルト 0.1 m**。物体は cm スケール（カメラから約 25 mm）
   なので丸ごとクリップされる。`clip_start` を物体サイズ基準の微小値にする必要がある。
3. **暗いワールド（0.02）＋単一バックライト**のみで、面を捉える光が乏しい。

→ 対策1（照明・クリップ）は必要だが、それだけでは透明物は見えない。
**決定打はハイブリッド（ガラス表面＋環境＋背景プロップ）**であることが実証された。

### 実証済み：ガラス表面は自作不要。Mitsuba エクスポートの exterior PLY を流用できる

`export mitsuba` は `exterior-000.ply`（物体外形の屈折界面。メートル単位、VDB と同一
座標系）を既に出力している。これを Blender に取り込み Glass BSDF（IOR 1.48）を割り当て、
背景に市松模様プレート＋床＋環境光を置いたところ、**透明テストピースが
「背景が屈折して透けるガラスブロック」として明瞭に描画された**（接地影付き）。
＝ B2 のハイブリッドは、VDB からの表面生成を待たずに **exterior PLY の再利用**で成立する。

この事実を受け、下記マイルストーンは「OpenVDB ボリューム単体」ではなく
**「Mitsuba exterior PLY をガラス表面、OpenVDB ボリュームを任意の内部媒質」**の
ハイブリッドを主線とする。

## マイルストーン（小さく刻む）

### B0. 黒画像の原因究明と最小デモの恒久化 ← **完了（2026-07-02）**
- 原因を実測で確定（詳細は `REDME_DEV.md` と上記「検証で判明した重要事実」）。
- デバッグ資産を恒久化済み：
  - `tools/phase0/inspect_vdb_grids.py` … 書き出したVDBの各グリッド値域を表示
    （「このピースは純透明レジン」等が即分かる）。
  - `examples/phase1/demo/blender_glass_demo.py` … Mitsuba export の `exterior-*.ply`
    をガラス表面にし、市松背景＋床＋環境光で1コマンド・レンダー。透明ピースが
    ガラスとして見えることを window_coupon / stepped_wedge で確認済み。
    `PIXELSTATS std` を出力し、std≒0 で「何も写っていない」退行を検知できる。
- 実行コマンドは `REDME_DEV.md` を参照。
- 注意：`examples/phase0/blender_cycles_volume.py`（ボリューム単体のpinnedプルーフ）は
  clip/照明の実バグを持つが、透明ピースでは原理的に見栄えしないため、demoは触らず
  上記 `blender_glass_demo.py` を主線とする。プルーフ側のclip修正は別途任意。

### B1. ステージ（撮影セット）を1つ手作りする ← **人間の手作業でOK**
- Blender で再利用シーン `demo_stage.blend` を作る。中身：
  - 環境光（HDRI 1枚。屋内 or スタジオ系）
  - 部屋 or 卓上の床＋簡単な壁、接地影が出るライティング
  - **背景プロップ**：模様のある花瓶／格子板／カラフルな小物を、
    カメラから見てオブジェクトの**真後ろ**に配置（透けと歪みが分かる位置）
  - 空の「オブジェクト配置用 Empty（アンカー）」を原点に置き、名前を固定（例 `VBDMAT_ANCHOR`）
  - カメラを固定し、名前を固定（例 `DEMO_CAM`）
- アセットは Blender 同梱 or CC0 素材でよい。凝りすぎない。
- 完了条件：`demo_stage.blend` を開くと、空のアンカー位置にガラス玉を仮置きしても
  「花瓶が歪んで透ける」画になることを目視確認。

### B2. VBDMAT オブジェクトの自動流し込み ← **AI 自動化（手続き的ステージ版は実質完了）**
- 手続き的ステージ（背景プレート＋床＋環境光）版は `blender_glass_demo.py` で既に成立。
  残タスクは、B1 の作り込んだ `demo_stage.blend` を使う版へ発展させること：
  1. `demo_stage.blend` を開く
  2. `exterior-*.ply`（ガラス表面）を読み込み、`VBDMAT_ANCHOR` に配置
  3. Glass BSDF（IOR は `scene-summary.json` / 既知値）を割り当て
  4. 任意で `exports/openvdb/volume.vdb` を内部媒質として重ねる（大半透明なので効果は限定的）
  5. `DEMO_CAM` から高解像度・デノイザ有効でレンダー → PNG 出力
- Docker は既存の `tools/Dockerfile.openvdb-cycles` を流用。
- 完了条件：1コマンドで window_coupon / stepped_wedge の“作品ショット”PNGが出る
  （手続き版は達成済み。`demo_stage.blend` 版が残り）。

### B3. Before / After 横並び ← **AI 自動化**
- 同じステージ・同じカメラで「プロセスモデル off（Phase 1 相当）」と
  「プロセスモデル on（Phase 2）」を2枚レンダーし、横に並べた比較PNGを1枚生成。
- 見せ方はキャプション付き（左：commanded voxels / 右：process-modeled）。
- 完了条件：普通の人が2枚を見比べて「右のほうが本物っぽい／馴染んでいる」と言える。

### B4. メインロードマップ追従の仕組み ← **AI 自動化**
- Phase 2 の各プロセスモデル（interface blur / droplet spread / mixing / …）を
  有効化した run の `optical.zarr` → `export openvdb` → B2/B3 が回るように、
  1本のドライバスクリプト（またはパイプライン設定）でつなぐ。
- 命名規約：`.local/demo/<sample>/<processmodel-tag>/beauty.png` のように残し、
  モデル追加のたびにギャラリーが増えていく形にする。
- 完了条件：新しいプロセスモデルを1つ足したとき、追加コマンド最小で
  そのモデルの Before/After ショットが生成できる。

## 明示しておく制約（誠実さのため）

- 係数は暫定・未校正。これは「校正済みの予測」ではなく**見た目の定性デモ**。
- Cycles のボリューム取り込みは内部 IOR 界面を落とす。屈折は外表面 BSDF で補っている
  ＝ 物理的に厳密ではなく「それらしく見せる」構成であることを出力に明記する。
- 物理忠実な検証が要る場面では Mitsuba 支線
  （`.local/sidemission_fance_demo_mitsuba.md`）を参照。
